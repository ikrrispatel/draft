# Draft — Deterministic Scoring System

## Overview

Draft uses a **deterministic, rule-based scoring system** to produce an Execution Readiness score for every blueprint. Scores are not generated by LLM judgment — they are computed from the presence, absence, and quality of specific fields in the shared state object.

Every score is justified with a machine-readable explanation referencing the exact rule(s) that contributed to it.

---

## Scoring Categories and Weights

| Category | Weight | What It Measures |
|----------|--------|------------------|
| Vision Clarity | 25% | Is there a single, clear wedge and a well-defined ICP? |
| Scope Discipline | 25% | Are there exactly 3 features and exactly 3 kill-list items, all aligned to the wedge? |
| Feasibility | 20% | Is the architecture achievable within the time constraint with the stated tech stack? |
| Distribution Readiness | 30% | Is there a concrete first-10 plan, an outreach message, and ICP-channel alignment? |

---

## Scoring Rules by Category

### Vision Clarity (25%)

| Rule ID | Condition | Score Effect |
|---------|-----------|--------------|
| V-1 | Wedge statement is present and non-empty | +30 points |
| V-2 | Wedge is a single proposition (no "and" connecting two distinct value props) | +25 points |
| V-3 | `vision.icp.role` is present and non-empty | +20 points |
| V-4 | `vision.icp.company_size` and `vision.icp.behavior_qualifier` are both present and non-empty | +15 points |
| V-5 | Wedge is flagged as unclear by QA (`qa.wedge_is_single == false`) | **Cap at 50** |
| V-6 | ICP is missing (`vision.icp.role` is null or empty) | **Cap overall score at 60** |
| V-7 | Wedge statement is missing | Score = 0 |

**Base:** Start at 0. Add points per rule. Apply caps last. Maximum: 90 (10 points reserved for QA consistency bonus — see QA section below).

---

### Scope Discipline (25%)

| Rule ID | Condition | Score Effect |
|---------|-----------|--------------|
| S-1 | Exactly 3 features present | +30 points |
| S-2 | Exactly 3 kill-list items present | +20 points |
| S-3 | Every feature has a non-empty `rationale` field | +15 points |
| S-4 | Every feature rationale references the wedge or ICP | +15 points |
| S-5 | Kill-list items are distinct from features (no overlap) | +10 points |
| S-6 | Feature count ≠ 3 | **Score = 40, trigger rerun** |
| S-7 | Kill-list count ≠ 3 | −20 points, trigger rerun |

**Base:** Start at 0. Add points per rule. Apply penalty rules last. Maximum: 90 (10 points reserved for QA consistency bonus).

---

### Feasibility (20%)

| Rule ID | Condition | Score Effect |
|---------|-----------|--------------|
| F-1 | Architecture `stack` is present and non-empty | +20 points |
| F-2 | Architecture `data_model` is present and non-empty | +15 points |
| F-3 | Architecture `api_surface` is present and non-empty | +15 points |
| F-4 | `fourteen_day_plan` is present with ≥ 3 phases | +20 points |
| F-5 | Plan fits within `intake.constraints.timeline_days` | +15 points |
| F-6 | API surface covers all 3 features (each feature has ≥ 1 endpoint) | +15 points |
| F-7 | Plan exceeds time constraint | −20 points |
| F-8 | Architecture section is entirely missing | Score = 0 |

**Base:** Start at 0. Add points per rule. Apply penalty rules last. Maximum: 100.

---

### Distribution Readiness (30%)

| Rule ID | Condition | Score Effect |
|---------|-----------|--------------|
| D-1 | `first_10_plan` is present and non-empty | +25 points |
| D-2 | `first_10_plan` specifies a channel | +10 points |
| D-3 | `first_10_plan` specifies a strategy with concrete numbers (e.g., "DM 10/day") | +15 points |
| D-4 | `first_10_plan` specifies a timeline within `intake.constraints.timeline_days` | +10 points |
| D-5 | `outreach_message` is present and non-empty | +15 points |
| D-6 | `outreach_message` is ≤ 300 characters (concise = usable) | +5 points |
| D-7 | `channel_icp_alignment` explanation is present | +10 points |
| D-8 | Channel matches ICP habitat (e.g., LinkedIn for B2B, Reddit for devs) | +10 points |
| D-9 | `first_10_plan` is missing | **Cap at 60** |
| D-10 | `outreach_message` is missing | **Cap at 60** |

**Base:** Start at 0. Add points per rule. Apply caps last. Maximum: 100.

## Hard Caps and Penalties — Summary

These rules override normal scoring and are applied **after all additive rules**:

| Trigger | Effect | Scope |
|---------|--------|-------|
| Missing ICP (`vision.icp.role` is null/empty) | **Overall score capped at 60** | Global |
| Wedge unclear (`qa.wedge_is_single == false`) | **Vision Clarity capped at 50** | Vision |
| Feature count ≠ 3 | **Scope Discipline set to 40**, rerun triggered | Scope |
| Missing outreach message | **Distribution Readiness capped at 60** | Distribution |
| Missing first-10 plan | **Distribution Readiness capped at 60** | Distribution |
| Architecture entirely missing | **Feasibility set to 0** | Feasibility |
| Wedge statement missing | **Vision Clarity set to 0** | Vision |

Caps are non-negotiable. They cannot be overridden by bonus points.

---

## Overall Execution Readiness Formula

```
Overall = (Vision × 0.25) + (Scope × 0.25) + (Feasibility × 0.20) + (Distribution × 0.30)
```

**After computing the weighted sum, apply global caps:**

1. If `vision.icp.role` is missing → `Overall = min(Overall, 60)`
2. If `qa.pass == false` (unresolved violations after rerun) → `Overall = Overall − 5`

The final score is an integer, rounded down (`Math.floor`).

## Score Explanation Rules

Every score must include a structured explanation. The explanation object follows this format:

```json
{
  "vision_clarity": {
    "score": 75,
    "rules_applied": ["V-1", "V-2", "V-3"],
    "rules_failed": ["V-4"],
    "caps_applied": [],
    "explanation": "Wedge is clear and singular. vision.icp.role is present but vision.icp.behavior_qualifier is empty."
  },
  "scope_discipline": {
    "score": 90,
    "rules_applied": ["S-1", "S-2", "S-3", "S-4", "S-5"],
    "rules_failed": [],
    "caps_applied": [],
    "explanation": "All 3 features and 3 kill-list items present. All rationales reference the wedge."
  },
  "feasibility": {
    "score": 80,
    "rules_applied": ["F-1", "F-2", "F-3", "F-4", "F-5"],
    "rules_failed": ["F-6"],
    "caps_applied": [],
    "explanation": "Architecture is complete and plan fits 14 days. One feature lacks a dedicated API endpoint."
  },
  "distribution_readiness": {
    "score": 85,
    "rules_applied": ["D-1", "D-2", "D-3", "D-4", "D-5", "D-7", "D-8"],
    "rules_failed": ["D-6"],
    "caps_applied": [],
    "explanation": "First-10 plan is concrete with numbers. Outreach message exceeds 300 characters — consider shortening."
  }
}
```

**Requirements:**
- Every explanation must cite the Rule IDs that were applied (passed) and failed.
- Every explanation must list any caps that were applied.
- The free-text `explanation` field must be ≤ 2 sentences and must reference a specific field or value from the shared state.

## How QA Affects Scores

### QA Consistency Bonus

Vision and Scope categories reserve 10 points (out of their maximums) for a **QA consistency bonus**:

| Condition | Bonus |
|-----------|-------|
| QA finds zero cross-agent conflicts for this category | +10 points |
| QA finds ≥ 1 conflict for this category | +0 points |

This means a perfect Vision score is 100 (90 from rules + 10 from QA bonus), and a perfect Scope score is 100 (90 from rules + 10 from QA bonus).

### Unresolved Violations

If `qa.pass == false` after the rerun cycle (i.e., violations persist):

1. The overall score receives a **−5 penalty** (applied after the weighted sum).
2. Each unresolved violation is logged in the score explanation with the Rule ID it violates.

### QA-Triggered Reruns and Score Recalculation

When QA triggers a rerun of an agent:

1. The agent's output is replaced in the shared state.
2. The scoring agent **re-scores only the affected category** using the updated output.
3. Other categories are not recalculated.
4. The overall score is recomputed from the updated category scores.

---

## Scoring Determinism Guarantee

To ensure identical inputs produce identical scores:

1. **No LLM involvement in scoring.** The Scoring Agent applies rules programmatically, not via prompt.
2. **Rules are evaluated in order.** Additive rules first, then penalty rules, then caps.
3. **Caps are applied last** and override any additive total.
4. **Rounding is deterministic:** `Math.floor` on the final overall score.
5. **Regression tests:** The `/examples` directory contains fixed intakes with expected scores. CI must validate these on every change to the scoring logic.
